How KAVACH Works

A complete system for proactive threat detection through intelligent honeypots and behavioral analysis.

Five Honeypot Token Types

Deploy different types of fake credentials throughout your infrastructure. Each one is a trigger point for attackers.

🔑 API Key

Fake API credentials that look real. Attackers will try to use them.

🌐 URL Token

Honeypot URLs embedded in code repos, internal docs, or backups.

📄 Document

Fake files with tracking tokens. Watermarked to catch leaks.

📧 Email Token

Honeypot email addresses in contact lists and mailing lists.

🔗 DNS Token

Subdomains that trigger alerts when queried.

Why this works: Attackers probe for credentials. They find your tokens. They have no idea they're fake until it's too late.

Eight Placement Strategies

1. Git Repositories

Commit fake API keys to public repos. Attackers will find them via GitHub dorking.

2. Internal Documentation

Plant tokens in wiki pages, runbooks, and internal knowledge bases.

3. Environment Files

Honeypot .env files on development machines and CI/CD systems.

4. Database Backups

Fake credentials in database dumps and backup files.

5. Code Comments

Tokens hidden in commented-out code that attackers might find during reconnaissance.

6. Configuration Files

Honeypots in nginx configs, Apache configs, and other server configurations.

7. Email & Chat Histories

Fake credentials referenced in email threads and Slack histories.

8. User Profiles & Metadata

Tokens embedded in user agent strings, comment threads, and metadata.

Risk Scoring

Each dimension contributes a score (0-100). The weighted sum determines the action:

  • 0-55: Allow (low risk)
  • 55-65: Flag (monitor)
  • 65-75: Challenge (require MFA)
  • 75+: Block (HTTP 403)
  • Honeypot Token: Score 99 (immediate block + alert)

Risk Thresholds & Actions

Example Attack Flow: Request arrives → Token detected in URL param ↓ Extract fingerprint: IP + Browser + OS + Device ↓ Classify: IP Reputation (known bad: +30), Request Rate (DoS-like: +20), Payload Analysis (SQLi: +25) ↓ Total Risk Score: 75 ↓ Action: BLOCK → HTTP 403 + Alert sent ↓ Log: Attacker profile created, event recorded, webhook sent to your team

Real-Time Profiling

Every attack triggers these steps:

  • Generate device fingerprint (MD5 of IP + User Agent + Language + Encoding)
  • Lookup or create attacker profile in database
  • Correlate with previous attacks by this fingerprint
  • Update threat score and behavior patterns
  • Send alert to configured channels (Slack, webhook, email)
  • Record full event with request details for forensics
